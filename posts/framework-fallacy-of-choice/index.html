Framework Fallacy of Choice

Feb 17, 2022 · 6 min read
  • Currently in my final semester of uni I am taking two courses in which I will use some sort of web framework
  • It’s a no brainer that Fullstack Web Development will use a framework, I just didn’t know that it would be predetermined
    • in this course we look at Vue and React
    • I have some experience with Svelte which seems a bit like Vue’s promising younger sibling. And as far as front end frameworks go I am extremely interested in Vue. However, it left me still lost as to the development of a full fledged backend / api.
  • In the past I used SvelteKit with their “endpoints” which is just a folder with js files that work as a backend.
  • I rather enjoyed this experience but it seemed very abstracted from an actual API, I didn’t feel like I was developing a web app. This is good and bad. In terms of development time and performance (Svelte is to my knowledge very performant). But I felt like I was too far from the actual API calls to really understand what is happening.
  • In my Databases Class we are tasked with a final project that interacts with a database in some way. This is pretty open ended, but I believe that the best way to accomplish this is with a web application. The reason being it can run on any machine with the installed dependencies much of the GUI work is taken care of with HTML, which is fairly simple something I know well.
  • Therefore I sought the best backend web framework to interact with a database.

So many languages and even more frameworks, which to choose? Javascript? Python? Go? Rust???

  • for my needs performance is not a requirement but is still a consideration.
  • I think off the bat I can exclude Rust. Although I like rust and respect anyone’s efforts to learn the language, it seems like the wrong choice for most web applications. I wouldn’t cut down a tree with a scalpel even if it is the sharpest tool in my repertoire. The syntax is a bit too long winded and for something that needs to be developed in roughly a month / prototyping rust is the wrong language.
  • Although I believe go is probably the best choice most of the time for web back ends the learning curve (thought slight) might cause problems with a project that needs to be engendered so quickly, and although I do plan on learning to create API’s with go, it isn’t as well flushed out should I run into some issues.
  • This leaves the battle of the “slow” languages (according to most programming memes). Although I enjoy JS and don’t consider it a bad language, in terms of backend frameworks it seems a little immature (comparatively speaking) and being no where near a JS wizard, living in the language for all aspects of my app doesn’t seem like a must. This leaves python. I chose python for two main reasons. Its ease of use with databases and its extremely mature backend frameworks.

Which Framework?

  • After choosing python the new choice of which python framework arises.
  • The “big three” are Django, Flask, and FastAPI.
  • Because of the simplicity of the front end and api for this database course I will go with flask
    • all that the “web” part of the project must do is render information from the database and handle input from one form for making queries. because of flask’s simplicity and the package flask-wtf this was extremely simple to get working.
    • using the jinja templates representing a simple “person” object was also very easy and allowed for a simple and decent looking front end for this project.
    • the meat and potatoes of the flask code can be seen below. it takes in preferences, passes data to a querying function and returns “matches” from a database with randomly generated people objects who have different attributes.
      # home
  @app.route("/")
  def home():
      """Render information on the project"""
      return render_template("home.html")


  # preferences
  @app.route("/preferences", methods=["GET", "POST"])
  def preferences():
      """Take in user preferences and return matches"""
      pForm = PreferenceForm()
      if request.method == "POST":
          # populate from pForm
          minAge = request.form.get("minAge")
          maxAge = request.form.get("maxAge")
          pName = request.form.get("pName")
          pCompany = request.form.get("pCompany")
          pField = request.form.get("pField")
          hName = request.form.get("hName")
          hKind = request.form.get("hKind")
          hSetting = request.form.get("hSetting")
          eName = request.form.get("eName")
          eType = request.form.get("eType")
          eMajor = request.form.get("eMajor")
          eDegree = request.form.get("eDegree")
          people = queryPrefs(
              minAge,
              maxAge,
              pName,
              pCompany,
              pField,
              hName,
              hKind,
              hSetting,
              eName,
              eType,
              eMajor,
              eDegree,
          )
          return render_template("matches.html", people=people)
      return render_template(
          "preferences.html",
          form=pForm,
      )
    • being able to get a working front end with such little code made flask an easy choice for this task.

Pure backend

  • a fun personal project I’ve been meaning to make for a while is a “scientific” workout app.

  • there are some basic equations for determining the weight you should be using, what your one rep max is, and whether or not your are working out within specifications given by prelipin’s chart (should you buy into it)

  • for this application I will be using a backend with a flutter mobile app front end.

    • although the entire app could be self contained in flutter I think that the skill of building and using an API makes it a fun and worthwhile.
    • additionally I think that having a server backend would allow for a web app because apple makes it very hard to deploy to their products rendering a full flutter app not applicable to everyone.

  • I think Django is my best bet for this project because

    • It is the oldest framework
    • great baked in tools for viewing the api without a frontend
    • it has the most “default” packages meaning that things adjacent to the main functioning of my project will be easy to get working (Templating, Url handling etc…)
    • It is the most popular which means that getting help will probably be the easiest.
    • The Django ORM allows for interaction with databases in the abstract meaning you can (more) easily migrate from one DB to another.
    • Testing is very simple, and with the Django shell it is easy to interact with the DB and with the Client class you can kindof simulate what a user would experience.
    • Scales very well
    • I want to make charts of previous workouts and python is great for handling / analyzing data
    • instagram uses a django backend so I know it can handle sending images back and forth with ease.

  • However Django does have some downsides

    • It is the slowest of the frameworks
    • it is the Django way or the highway
    • it abstracts some things that I would like to learn

  • But for my use case Django is the best fit.
