Is emacs worth the strife?

May 20, 2021 · 4 min read

Why use Emacs (Doom/ Spacemacs / Vanilla)

  • Since I started studying Computer Science I have had to work with several different languages / frameworks and as a result have had to install a myriad of applications to handle each new challenge and or language.
  • I am not a fan of application sprawl as I don’t like clutter, using up extra hard drive space, or having to learn how a new program works.
  • if you have one application that can do every (or at least most) of the daily tasks that you have your work-flow will be constant and there will be one large learning curve followed by miniature additions here and there versus re learning a new system for every program.
  • I am sure that there are many more things that some of these programs can do but in my work flow this is how I had my work divided before switching to emacs.

I use vim for flutter but wouldn’t consider it an application in the same sense as the others as it is a CLI tool and the keybindings used transfer over to emacs / vs code

  • on a more “moral/philosophical” level I think there is a great advantage building and fully understanding the tools that you use. The beauty of a lathe is that it can create all the parts used to make a lathe (or so I have been told). I think it is the same with emacs. I saw this with vim and really loved the process of building out my own environment. Emacs takes this to another level with the language homogeneity. Everything is an elisp file, everything is a series of functions that work to add functionality. This is what made me cross the picket line into the world of emacs.
  • I’m not sure if I’ll ever fully replace vscode as it has some IDE like functionality that I need in the immediate future. My goal is to use it less and less until I am able to fully replace it if possible.

Vanilla vs Doom

  • I started my emacs journey with doom emacs. I really enjoyed using it but it felt like a “cooler” vs code. It was more hackable and much more involved than vs code but still it felt too premade. As I stated previously my goal with branching out into the “do it yourself” editors was to do it myself. I felt like I was using someone’s emacs config and wanted to design my own, even if it was a bit less streamlined. If my main focus was purely on productivity then I would most likely stick with doom, it is extremely well made and curated especially for programmers. However, with the quarantine keeping me home more than I’d like I figured that I could use the extra time to try and make my own config.

Vim vs Emacs

  • I started using vim for the same reasons as emacs. I wanted to build something from the ground up. However, there was an extra factor that brought me to vim, its efficiency. I would say vim (or in my case neovim) can be broken into two categories. The first is a set of highly effective and ergonomic key-bindings and the second is a very nice and extensible terminal text editor. I am a huge fan of both but were it not for the key-bindings vim would not be half as valuable.

  • when I found out about evil mode I decided to give emacs a try. It seemed like it was a proper version of part of vim, and one that was well integrated with a different editor. I don’t think you could call emacs a text editor but vim definitely is, and being able to use this text editor within emacs fills the gaps of vim and emacs (in my opinion).

  • In a similar sense I consider Vs code one the best IDE’s available and when it is paired with vim key-bindings it becomes a great IDE with a great text editor.

  • All in all I think that if you want to fully understand the tools that you are using then emacs and CLI vim are very important, and in many cases fun (if you enjoy configuring a system to your liking). Even if you are not one of those people at the very least learning the vim key bindings with something like VS code will greatly increase your productivity and make editing text a breeze.
