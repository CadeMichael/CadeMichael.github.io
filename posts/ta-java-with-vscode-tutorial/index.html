TA Java With Vscode Tutorial

Sep 21, 2021 · 5 min read

Dr java vs vs code

  • I know that Connamacher wants us using DrJava and I recommend this for most students. However DrJava is relatively old and incompatible with most java distributions. This means that it takes a great deal of finagling just to get it working. It is frustrating that java8 is the last supported version of java for DrJava and most “distributions” of the jdk are incompatible. This leaves you as students of this class with several options
  1. Use Myapps.case.edu for a virtual desktop if you can deal with the slowness. For absolute beginners who have little to no interest in seeing “what’s under the hood” this is your best bet.
  2. Try to get dr DrJava working on your machine. For the best results I’d say use linux and if you are not comfortable with linux windows. It seems DrJava doesn’t want to work on Mac Os.
  3. Go for the full IDE experience. If you already know java / plan to work with it this isn’t a bad idea but for beginners the lsp functions like autocomplete and linting can be crutches that will leave you in bad shape for the tests.
  4. Use a simple text editor and the command line. This is what I would suggest to students who plan to major in CS. You will gain valuable experience in working with the command line and gain experience using a great text editor. Personally I would recommend vim but this is a huge adjustment that might be hard to make just for this class. That leaves vs code, I think this is a great entry point for your CS career as vscode has a vim mode and can be used with almost any language so you won’t have to learn it for this class and never use it again. This tutorial will specifically cover how to get setup for Mac Os and Linux the two best OS’s for programming in ascending order (Give linux a try!). However, the resources for this will provide instructions for windows, and you can use vscode with wsl on windows.

Getting the jdk & editor

  • for this task I cannot recommend a package manager enough. On Mac os this is homebrew and for linux it is the package manager of your distribution. Using a package manager allows for “tidy” packages. They are installed and uninstalled for you without leaving behind junk files. it also ensures that you are getting safe and recent packages, that can be easily upgraded.

Mac OS

  • you might have to enable “developer settings”, I guess this means mac’s aren’t for developers anymore.
  • commands to run in the terminal, commands are the lines followed by $, # marks comments
$ brew install coretto8 # install java
$ java -version # check the version, should return the following 
openjdk version "1.8.0_302"
OpenJDK Runtime Environment Corretto-8.302.08.1 (build 1.8.0_302-b08)
OpenJDK 64-Bit Server VM Corretto-8.302.08.1 (build 25.302-b08, mixed mode)
$ brew install visual-studio-code # install vs code

Linux if you’re using linux this should be a piece of cake

$ sudo pacman -S openjdk8 # pacman is the package manager for arch, your's will be different if you are on a differnt linux distro 
$ java -version # check the version, should return the following 
$ sudo pacman -S code # in linux vs code is called code

Getting Junit

  • Junit is something we will use later to run tests.
  • follow the resource links to download the respective jar files and put in your .bashrc or .zshrc
  • I have the jars in a directory called javaTA, you can put this wherever you want so long as your JUNIT_HOME path reflects it.
# java 
export JUNIT_HOME="$HOME/Desktop/javaTA"
export PATH="$PATH:$JUNIT_HOME"
export CLASSPATH="$CLASSPATH:$JUNIT_HOME/junit.jar:$JUNIT_HOME/hamcrest-core.jar"

using JUNIT

  • to get Junit we have to import it in the header of our test files and run a special command from the terminal.
import org.junit.*;
import static org.junit.Assert.*;
  • then when running your tests simply run
java org.junit.runner.JUnitCore [name of tester class]

using vs code with java

  • for this please resist using a linter or lsp. we will compile with the terminal command javac and run with java.
  • ex
    • If we have one file and the class name is Test, then the file should be named Test.java. We would compile this one file with…
public class Test {
	  public static void main(String[] args) {
		  System.out.println("hello world");
	  }
}

$ javac Test.java # compiling test
$ java Test # running test
hello world
  • for multi file projects there can be a main function in only one class and although all files will need to be compiled that is the only one that will need to be run.
  • compile a multi file project like this
# either cd into the directory and compile from there
$ cd projectdir
$ javac *.java # this compiles all files in the dir with .java endings 
# compile all files in the project directory from the outside 
$ javac projectdir/*.java
